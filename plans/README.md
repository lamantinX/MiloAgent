# Implementation Plans

Generated by the improve skill on 2026-07-15. Execute in the order below unless dependencies say otherwise. Each executor must read the selected plan fully, honor its STOP conditions, run every verification gate, and update the status row when done.

The target model is deliberately explicit:

- A **business** is an isolated workspace/tenant.
- A **product** is the existing project configuration and belongs to one business.
- A **service account** belongs to exactly one business and may be assigned to one or more products inside that business.
- A business may own multiple distinct accounts for the same service.
- Automated and manual actions must never fall back to an account from another business.

## Findings considered and rejected

- Reusing `project` as the new business identifier was rejected because the requested model needs businesses to contain products and accounts independently.
- A process-global “active business” was rejected because browser sessions and background jobs would overwrite each other’s context.
- Sharing one external account across businesses was rejected by default because it creates data, persona, and credential leakage. Explicit transfer can be added later as an audited operation.
- Using Telegram Bot API for group comments was rejected. Bot API remains limited to the admin/control dashboard; group engagement uses authenticated Telethon user sessions.
